## ⚠️ 铁律声明

**CRITICAL — 触发后第一个动作必须是：执行阶段0+阶段1（前置判定+标准收集）。NEVER跳过阶段1直接进入执行。原因：没有标准的Loop跑完也是白跑。**

**CRITICAL — 触发条件严格：只有用户以"循环""loop"开头的话才进入Loop模式。没有这个前缀一律按正常对话处理。原因：防止普通问题误触发导致体验变差。**

**CRITICAL — 自查和子Agent审查是后台自动完成，NEVER让用户参与中间过程。用户只在开头给标准和结尾收结果。原因：用户中途参与=回到传统"你一句我一句"模式，Loop失去意义。**

---

## 功能定位

本Skill做一件事：**把"你一句我一句"的交互模式，切换成"你说一次，我在后台闭环跑到达标"的模式。**

你只需要参与两个环节：
1. **首次某类任务时**给标准（我给选项让你选，一次性的）
2. **结尾收结果**（看一眼执行履历，ok就过）

中间所有"执行→自查→子Agent审查→修正→再检查"的循环，都在后台自动跑。

## 触发条件

### 应该触发
- 用户以 **"循环 "**（循环+空格+任务描述）开头 — 如："循环 帮我审晚安眼罩详情页"
- 用户以 **"loop "**（loop+空格+任务描述）开头 — 如："loop 帮我生成竞品分析报告"
- 用户说完整触发短语："循环工程"、"loop工程"、"自主跑"
  原因：只用"循环"前缀会误触"循环播放""循环经济"等非命令式语义。"循环+空格"限定为命令式入口

### 不应该触发
- 用户说话不以"循环""loop"开头 — 按正常对话处理
- 纯闲聊、简单问答、一句话能搞定的小事
- 用户说"直接给我结果"或"不用循环"等明确拒绝Loop模式的表达

## 🔒 防跳步机制

本Skill使用以下机制：
- ✅ 步骤产出物锁（每步📦）
- ✅ 思考检查点（关键节点🧠）
- ✅ 熔断器（外部调用🔌）
- ✅ 完整性锁（输出前🔒）
- ✅ 卡死检测（产出无实质进步时自动终止）
- ✅ 文件异常恢复（JSON损坏→重新收集标准；progress丢失→从头开始标记为第1轮）

## ⚡ 平台兼容

**子Agent可用性判定（每次启动时自检）：**
- 如果可以调用 Agent 工具 spawn 子Agent → **完整模式**（执行+子Agent审查+修正+再审查）
- 如果 Agent 工具不可用（其他平台如Trae/Qoder）→ **NoAgent模式**（执行+自查+修正+再自查）
- NoAgent模式下，子Agent审查降级为自查清单逐项核对+额外一致性检查

| 契约层 | 定义 |
|:---:|------|
| **输入契约** | 用户以"循环/loop"开头的任务描述 + 首次某类任务时用户选的标准 |

- **是** → 进入Loop模式，输出 `Loop Engineering Loop模式已启动`

### 🔌 熔断器

### 阶段1：标准收集（首次或模糊时触发）

- 任务类型匹配 + 检查清单条目完全一致 = 同类任务，直接复用

- 文件缺失或损坏 → 走初始化流程（问标准）

```
🧠 思考检查点
```

### 🔌 熔断器
- 成功条件：用户清晰选择了每项标准
- 失败处理：用户说"你看着办" → 优先按 JSON 中该任务类型的预设标准执行；若JSON无匹配则使用默认标准（平衡模式+3轮）
- 重试次数：对模糊回答追问1次，仍模糊→默认标准
- 降级方案：用户中途退出 → 标记任务待处理

### 阶段2：循环执行（闭环核心）

📦 产出物：progress.md（存放在当前工作目录下，文件名固定为 progress.md，单线程不冲突）+ 本轮产出

循环结构如下——每一轮自动执行，你全程不在场。每轮迭代的边界是：**子Agent给出审查结果**。

```
【第N轮】从执行开始，到子Agent给出审查结论结束

  │
  ├─ ① 读进度记录
  │    读取 progress.md（第一轮不存在则跳过）
  │    对上轮反馈的问题逐条确认，防止遗漏
  │    
  ├─ ② 执行本轮工作
  │    ├─ 第1轮：按你给的标准从头做
  │    └─ 第N轮（N>1）：基于上轮子Agent的反馈修正
  │        如果触发过"策略切换"（反疲劳标志位=ON）：
  │          换方法修，不用之前试过的方式
  │    
  ├─ ③ 自查：对照检查清单逐项核对——**必须逐项列出不可笼统说"通过"**
  │    格式示例（写入 progress.md）：
  │    - [x] 广告法合规：全文无违禁词
  │    - [ ] 产品信息准确：规格待核对
  │    ├─ 全部通过 → 下一步
  │    └─ 有未通过 → 修 → 再自查（内部循环，不占最大轮次计数——原因：自查是强制质量门禁，不应消耗外部轮次额度）
  │        内部循环最多3次（硬性安全限制，不可配置——原因：防自查陷入死循环消耗Token）
  │        → 仍不过 → 将未通过项实情记录到progress.md → 跳下一步（子Agent会审查这些未通过项）
  │    
  ├─ ④ 审查（子Agent可用时spawn，不可用时自检替代）
  │    **子Agent可用？** → 用 Agent 工具 spawn 审查子Agent（指令如下）
  │    **子Agent不可用（NoAgent模式）？** → 降级为自查清单逐项核对+额外一致性检查
  │    子Agent指令（按下述生成，仅子Agent模式使用）：
  │      "你是一个独立的审查Agent，只读不可修改。
  │       请按以下标准审查[任务类型]的产出：
  │       - 严格度：[用户选的严格度]
  │       - 检查清单：[用户选的检查项]
  │       - 找茬视角：假设这个产出要发布，你会找出什么问题？
  │       审查结果格式：通过/不通过 + 具体问题清单（每条问题标注严重程度）"
  │    ├─ 审查通过 → 本轮达标，准备终止交付
  │    └─ 审查不通过 → 分析反馈 → 记录进下一步
  │        ⚠️ **审查不通过后修正的产出必须进入下一轮重新审查**——修正后直接交付违反"执行与审查分离"原则
  │
  ├─ ⑤ 反疲劳判定（决定下一轮的修改策略）
  │    对照 progress.md 中的上轮子Agent反馈：
  │    ├─ 首次不过 → 正常修正策略 → 反疲劳标志位=OFF
  │    └─ 同一问题连续两次不过 →
  │        判定标准：子Agent连续两轮反馈指向同一代码位置/文档段落/数据字段，
  │        或反馈核心关键词高度一致（即使措辞不同）。无法判断时视为同一问题。
  │        触发后：日志记录"策略切换" → 反疲劳标志位=ON
  │        → 换方法修（不要重复相同路径）
  │    
  ├─ ⑥ 写入执行日志 → progress.md
  │    ├─ 本轮做了什么
  │    ├─ 子Agent发现了哪些问题
  │    ├─ 本次修正了哪些
  │    ├─ 反疲劳标志位状态（ON/OFF）
  │    └─ 本轮产出与上轮对比：
  │       ├─ 有实质进步 → 循环继续 → 下一轮
  │       └─ 无实质进步（见下方卡死判定）→ 判卡死，跳到阶段3终止
  │
  └─ ⑦ 🧠 思考检查点（每轮终结点）
        检查结果写入 progress.md（追加在每轮日志之后）：
        本轮Token预估消耗：
        子Agent反馈质量评估：
        反疲劳触发状态：
        本轮进步置信度：
        预算是否仍在可控范围：
```

**卡死判定标准（按任务类型——子Agent反馈为主要依据，量化阈值仅供参考）：**

| 任务类型 | 判卡死条件 |
|---------|-----------|
| 审核文案 | 本轮改动极为微小（肉眼难见变化）且 子Agent发现问题数未减少 |
| 生成文档 | 新增内容微乎其微 且 检查项未通过数未减少 |
| 分析数据 | 数据产出近乎不变 且 质量分数未提升 |
| 修复代码 | diff行数极少 且 问题数未减少 |

> 若无法可靠量化——以子Agent反馈为唯一卡死依据：**子Agent连续两轮说"跟上一轮没差别"即判卡死。**

### 🔌 熔断器（自查内部循环）
- 成功条件：自查清单全部通过
- 失败处理：自查未通过 → 修正 → 重试（内部循环，不占外部轮次计数）
- 重试次数：最多3次（硬性安全限制——原因：防自查陷入死循环消耗Token）
- 降级方案：3次内循环后仍未全过 → 实情记录当前状态 → 跳子Agent审查（由子Agent判断是否可接受）

### 🔌 熔断器（审查环节-子Agent模式）
- 成功条件：子Agent返回"通过"
- 失败处理：子Agent返回"不通过" + 具体问题清单 → 我分析问题后修正
- 重试次数：等于你设定的最大轮次 - 当前已用轮次
- 降级方案：子Agent持续无法通过且轮次用尽 → 跳阶段3超限终止

### 🔌 熔断器（审查环节-NoAgent模式）
- 成功条件：自查清单逐项核对全部通过 + 额外一致性检查无问题
- 失败处理：自查项不通过 → 修正 → 下一轮
- 重试次数：等于你的最大轮次 - 当前已用轮次
- 降级方案：轮次用尽 → 跳阶段3超限终止

### 🔌 熔断器（子Agent spawn 失败）
- 成功条件：子Agent正常返回审查结果
- 失败处理：子Agent spawn 失败或超时 → 降级为我自检+自查清单核对
- 重试次数：1次
- 降级方案：自检替代，如实记录"子Agent未启动，改用自检"

### 阶段3：终止判定与交付

📦 产出物：最终交付物 + 执行履历

每次循环结束后检查终止条件，遇到以下任一情况即终止：

| 状态 | 触发条件 | 怎么做 |
|:---:|---------|--------|
| ✅ **达标交付** | 子Agent审查通过 | 交付最终结果 + 执行履历（见下方模板） |
| ⛔ **超限终止** | 达到你设定的最大轮次 | 交付当前版本 + 说明哪些检查项没通过 + 建议下一步 |
| 🚫 **卡死终止** | 本轮产出跟上轮无实质进步（≥90%相似） | 交付当前版本 + 说明卡在哪 + 建议 |
| 🛑 **用户中断** | 你说"停"或"不用循环了" | 立即停，交付当前进度 + 已完成的部分 |

**执行履历模板**（随每次交付附带）：

```
📦 交付物: [文件名/路径]
📋 执行记录:
  第1轮: [做了什么] → 子Agent发现[问题X] → 未通过
  第2轮: [修正了Y] → 子Agent发现[问题Z] → 未通过
  第3轮: [修正了Z] → 子Agent通过✅
  总轮次: 3 | 策略切换: 1次 | Token消耗估级: 高/中/低
```

### 🔒 完整性锁
📦 产出物：最终交付物 + 执行履历
```
🔒 完整性锁
- [ ] 交付物已生成
- [ ] 执行履历已附带（谁、做了啥、通过了没）
- [ ] 如非达标交付，说明了卡在哪/哪项未通过
- [ ] 如策略切换，记录了切换原因
- [ ] 执行履历中的轮次数与实际执行步骤一致
- [ ] 本轮任务类型+标准已写入 references/task-standards.json
```

---

### 阶段4：后续处理

📦 产出物：标准写入 references/task-standards.json + progress.md 保留

**标准自动持久化**（每次交付前执行）：
- 本轮任务类型 + 检查清单 + 严格度 + 最大轮次 → 写入 references/task-standards.json
- **即使是"自定义标准"也要解析后写入**——拆解后的检查子项一并存入
- 存储规则：同一任务类型只保留最新一条标准，覆盖旧配置

```
提交交付物后：
  ├─ 你看结果，说"OK" → 本轮任务结束，标准已保留
  ├─ 你看结果，说"再改一个点" → 不进入Loop，直接改（单次修改）
  └─ 你看结果，说"这次标准漏了XX，下次加上" →
         第1步：确认追加/替换
         第2步：列出当前标准 → 用户确认修改项
         第3步：写入 references/task-standards.json（覆盖旧配置）
```

## 判断标准

| 状态 | 定义 | 怎么做 |
|:---:|------|--------|
| ✅ 正常 | 子Agent通过（完整模式）或自查全过（NoAgent模式） | 正常交付，无需用户额外操作 |
| 🟡 警告 | 子Agent未启动→NoAgent模式，或策略切换触发 | 执行履历中如实记录降级/切换情况 |
| 🔴 错误 | 审查连续失败且轮次用尽 | 交付时标注"审查精度受限"并说明未通过项 |

## 禁止行为

| # | 禁止 | 原因 | 正面替代 |
|---|------|------|---------|
| 1 | NEVER在一次用户对话后立刻交付未自检的结果 | 原因是Loop的核心价值就是闭环自迭代，跳步骤=没用 | 必须完成自查→子Agent审查→修正闭环后才交付 |
| 2 | NEVER让用户参与自查或子Agent审查环节 | 原因是用户只在开头定标准和结尾收结果，中间不打扰 | 自查和子Agent审查全程后台进行，只向用户交付最终结果+履历 |
| 3 | NEVER在子Agent审查不通过时不分析直接重跑 | 原因是盲目重跑=碰运气，要靠分析反馈后的精准修正 | 子Agent不通过后先分析反馈→定位根因→针对性修正→再提交审查 |
| 4 | NEVER设置无限最大轮次 | 原因是Token成本失控是Loop工程最大的坑 | 最大轮次必须用户确认或默认为3轮，到达后超限终止 |
| 5 | NEVER默认处理新任务类型时不问标准 | 原因是没标准=闭环没有检查清单=无法判断达标 | 首次碰到某任务类型先问一轮标准（选项式），之后复用 |
| 6 | NEVER在未读progress.md的情况下开始新一轮 | 原因是不知道上轮做了什么，可能重复做或遗漏问题 | 每轮开始前先读progress.md，对上轮反馈逐条确认 |
| 7 | NEVER在尚未完成标准收集的任务上运行子Agent审查 | 原因是子Agent不知道按什么标准审，审了也白审 | 子Agent审查前先确认该任务类型的检查清单已在阶段1中收集完成 |
| 8 | NEVER在子Agent不通过后修正产出不进入重新审查就交付 | 原因是修正后直接交=自己改自己查=循环断链 | 修正后的产出必须进入下一轮子Agent审查，通过后才能交付 |
| 9 | NEVER在阶段4跳过将标准写入task-standards.json | 原因是不写JSON下次同类任务不能自动复用标准 | 每次交付前必须将本次任务的标准写入JSON |
