# 蔡氏福宁文案合规审核

## 概述

帮助审核人检查蔡氏福宁品牌文案是否符合广告法要求，同时支持背面标签的产品信息质检（逻辑/错别字/标点）。

**输入格式**：支持文字、图片（PNG/JPG）、PDF、DOCX  
**品类定位**：日用品（养生个护：眼罩/暖贴/贴敷/泡脚/助眠系列）  
**审核原则**：谨慎使用（需要有一定依据或行业惯例支撑）  
**品类标准**：统一标准（各品类共用一套词库）

## ⚙️ 前置条件（团队安装前必读）

本 skill 在审核流程中会调用**企业微信文档** 读取法务背书词汇清单。新用户使用前需要完成以下配置：

### 1. 确认企微 CLI 已安装

运行以下命令确认：
```bash
which wecom-cli && wecom-cli --version
```
如未安装，联系锴哥获取安装包。

### 2. 确认企微机器人已授权「文档」权限

运行测试命令：
```bash
wecom-cli doc create_doc '{"doc_type": 3, "doc_name": "测试文档"}'
```
如返回 `errcode: 851014`（authorization expired），需重新授权：
> 企业微信 → 工作台 → 智能机器人 → 找到对应机器人 → 授权管理 → 开启「文档」权限

### 3. 确认企微机器人已授权「获取成员文档内容」权限

运行测试命令：
```bash
wecom-cli doc get_doc_content '{"docid":"<某个docid>","type":2}'
```
如返回 `errcode: 851008`（partial no authorization），需再次授权，勾选「获取成员文档内容」。

> ⚠️ **团队成员协作说明**：法务背书词汇清单托管在企业微信文档，**所有团队成员均可直接编辑**，无需联系锴哥手动录入。

## 🔍 核心流程（四步制）

> **设计原则**：每步有明确输入→输出→人工确认点。上一步输出确认后才进入下一步。
> AI在每个步骤末尾停下来等待审核人确认，不自动跳到下一步。

---

### 步骤1：定性 — 确定审核范围

**目标**：搞清楚"审什么、用什么标准审"

**收到文案后，先问审核人**（直接复制发送）：
```
收到，我先确认几个信息：

1. 文案类型？（包装/电商主图/电商详情/线下展架/线下DM单/小红书/抖音/公众号）
2. 什么产品？哪个系列？
3. 有没有特别想重点看的地方？
```

**根据回答确定审核维度**：

| 文案类型 | 监管强度 | 审核维度 | 触发企标？ |
|---------|----------|---------|-----------|
| **包装**（正面+背面） | 🔴 最严 | 广告法 + 标签质检 + 企标对照 | ✅ |
| **电商主图/首图** | 🔴 最严 | 广告法（标准≈包装正面） | ❌ |
| **电商详情页** | 🟡 中等偏严 | 广告法（功效暗示可加限定词） | ❌ |
| **线下展架/POP** | 🟡 中等 | 广告法（字少功效词更敏感） | ❌ |
| **线下DM单/宣传册** | 🟡 中等偏松 | 广告法（偏消费者教育） | ❌ |
| **小红书** | 🟢 宽松 | 广告法（⚠️ 截图传播快） | ❌ |
| **抖音** | 🟢 宽松 | 广告法（⚠️ 口播评论区不可控） | ❌ |
| **公众号** | 🟢 宽松 | 广告法（⚠️ 长期留存链接） | ❌ |

**步骤1输出**（确认后进入步骤2）：
```
📋 定性结果：
- 文案类型：[类型]
- 产品：[名称+系列]
- 审核维度：[列出要审的维度]
- 重点关照：[审核人指定的关注点，或"无"]

确认后我开始提取内容。
```

> ⚡ **预加载优化**：步骤1确认后，AI 在进入步骤2的同时，**后台异步启动法务词汇读取**（调用步骤3a 的 `wecom-cli doc get_doc_content` 轮询）。步骤2 和步骤3a 并行推进，到步骤3a 时法务词汇大概率已就绪，无需等待。

---

### 步骤2：提取与核实 — 确保数据准确

**目标**：把文案内容完整、准确地提取出来，经审核人确认后作为审核基础

> ⚠️ **铁律：识别失败，严禁脑补**
>
> 如果无法正确读取文案内容，AI必须如实告知审核人，严禁自行猜测编造。
> 基于错误内容审核 = 审核结果无效。此规则优先级高于任何输出要求。

#### 2a. 双重读取

用**两种方式**读取同一文件，对比结果是否一致：

| 文件类型 | 方式A（主读） | 方式B（核对） |
|---------|-------------|-------------|
| 文字（直接发） | 直接读取 | 无需对比 |
| 图片（PNG/JPG） | 多模态识别 | 多模态再次读取 |
| DOCX | python-docx 提取 | 多模态读取文件 |
| PDF文字型 | pdfplumber 提取 | 多模态读取文件 |
| PDF图片型/设计稿 | 转图片 + 多模态读取 | 多模态再次读取 |

> PDF设计稿：先用pdfplumber试读，文字极少（<10汉字）/乱码/报错 → 自动判为图片型，转图后读取。

**前置过滤**（防止乱读进入对比）：

| 过滤项 | 标准 | 处理 |
|--------|------|------|
| 总字数 | ≥ 30个有效汉字 | < 30 → 无效，切换方式 |
| 乱码率 | 非汉字/非标点占比 < 15% | ≥ 15% → 无效，切换方式 |

**逐字对比**：提取全部有效中文字符序列（去标点、空格、换行），逐字对照。

- **完全一致** → 进入 2b
- **有差异** → 报告差异点，审核人确认哪个版本正确后，进入 2b

#### 2b. 内容核实与初筛（⚠️ 不可跳过）

> **为什么必须做**：AI视觉识别会混淆数字（6↔5、3↔8、1↔7）、误读产品类型、漏读成分；人工设计也会出现错别字、逻辑矛盾、前后不一致。
> 一次确认10秒，修复错误审核报告要10分钟。

提取关键内容 + 初筛文案本身可能存在的问题，**逐条报给审核人确认**：

```
📝 内容核实（请逐条核对，确认无误后我开始审核）：

【关键数据】
1. 产品名称：[名称]
2. 规格/片数：[X片/盒 或 尺寸等]
3. 成分列表：[成分1、成分2、成分3...]
4. 产品类型：[如：蒸汽热敷贴/暖贴/眼罩等]
5. 执行标准号：[标准号]
6. 委托方：[公司全称]
7. 受托方/生产厂：[工厂全称]
8. 保质期：[时长]
9. 其他关键数字：[温度/尺寸/含量等，如没有则不列]

【初筛问题】（文案本身可能存在的问题，所有疑似问题必须列出，宁多勿漏）
10. 错别字/漏字：[如有，列出具体位置和内容；如没有则写"未发现"]
11. 标点符号异常：[如有，列出；如没有则写"未发现"]
12. 逻辑/前后矛盾：[如有，列出具体矛盾点；如没有则写"未发现"]
13. 格式不统一：[如规格写法前后不一致、品牌名中英混用等，列出；如没有则写"未发现"]
14. 其他可疑之处：[如有，列出；如没有则写"未发现"]

以上是否准确？有误请指出。
```

**规则**：
- 不确定的标记为"⚠️ 未能清晰识别"
- **所有疑似问题必须列出，宁多勿漏**——但凡有一丝不确定的都列出来让审核人判断，AI不要自行过滤
- **初筛问题不等于审核结论**——这里只做"有没有问题"的初筛，风险的等级和修改建议留到步骤3
- 审核人指出错误 → 重新读取 → 再次确认 → 直到全部正确
- 审核人确认全部无误 → **干净数据确认 ✅** → 进入步骤3

**步骤2输出**：审核人确认后的完整干净数据，作为步骤3的唯一输入。

### 🧠 思考检查点（步骤2完成时）

确认干净数据已就绪后，在进入步骤3前执行五维检查：

```
🧠 思考检查点
├─ 数据完整：[步骤1的定性信息已确认？步骤2的干净数据已全部确认？]
├─ 执行合规：[步骤1→步骤2→步骤3，没有跳步？]
├─ 安全合规：[数据中没有混入禁止内容？无涉密信息泄露风险？]
└─ 预算合理：[审核任务量是否适合一次完成？是否需分块处理？]
```

### 步骤3：审核判断 — 逐项检查风险

**目标**：基于步骤1确定的维度 + 步骤2确认的干净数据，逐项审核

#### 3a. 法务背书词汇匹配

> 每次审核前，从**企业微信文档**读取最新法务背书词汇清单。

**读取时机**：

| 场景 | 做法 |
|------|------|
| 步骤1完成后已启动预加载 | 检查轮询结果：已就绪 → 直接用；未就绪 → 走下方轮询流程 |
| 步骤1完成后未预加载 | 立即按下方方式开始读取 |

**文档信息**：
- 文档名称：`蔡氏福宁 违禁词白名单`
- 文档 URL：https://doc.weixin.qq.com/doc/w3_AP4ARniGAO0CNaLYbhWESQZGo8MTS_a?scode=ACIASgcLAAwukdZvUEAP4ARniGAO0
- `docid`：`dce_tRfn7KasEa5-jt0R9m9-VBhCrZtoyuoAyOFzwzwTdFt2vQ_WMSXYfU8HbfaCtB9h4DWuQtSz-1fzEcaUgAcw`


**读取失败处理**：
- ⚠️ **不得回退到硬编码清单！**
- 告知审核人具体错误，提供两个方案：
  - **检查企微授权** → 参考本 skill「前置条件」章节，确认：① `wecom-cli --version` 正常；② 机器人已授权「文档」+「获取成员文档内容」权限
  - **手动提供清单** → 紧急情况下，审核人临时提供文本格式词汇清单

**匹配规则**：
- 找到已背书词汇 → 在报告中列出 + 注明背书依据 → 不跳过后续审核
- 未找到已背书词汇 → 正常进入审核

#### 3b. 按维度审核

**维度A：广告法合规**（所有文案类型都审）

| # | 审核项 | 检查内容 |
|---|--------|---------|
| 1 | 涉医暗示 | 医疗用语、疾病治疗、疗效承诺 |
| 2 | 虚假宣传 | 无依据的功效宣称 |
| 3 | 绝对化词 | 最好/第一/顶级/必备 |
| 4 | 数据宣称 | 陈年/百分比/功效数据是否有依据 |
| 5 | 违禁词 | 参考 `references/违禁词与边界词清单.md` |

同时执行**打假风险三步自检法**：
- Step 1：标红排查 — 过一遍所有形容词/副词、极限词
- Step 2：证据链核查 — 功效/数据/对比是否有依据
- Step 3：降级替换 — 高风险词找安全替代

**风险分级**：

| 等级 | 判断标准 | 处理 |
|------|----------|------|
| 🔴 高风险 | 明确违法/涉医明示/绝对承诺 | 必须修改 |
| 🟡 中风险 | 擦边/暗示/灰色地带 | 风险提示，审核人决定 |
| 🟢 低风险 | 不精准/有优化空间 | 建议修改 |

**维度B：标签合规检查**（⚠️ 仅包装背面，纯合规判断）

> 步骤2b已处理错别字/标点/逻辑矛盾等内容准确性问题。维度B只审**标签要素是否齐备、格式是否合规**。

**标签必含要素检查**（逐项勾选）：

| # | 必含要素 | 审核内容 |
|---|---------|---------|
| 1 | 产品名称 | 是否标注 |
| 2 | 成分表 | 是否完整列出 |
| 3 | 规格/数量 | 是否标注且与正面一致 |
| 4 | 厂商信息 | 委托方+受托方名称、地址、联系方式 |
| 5 | 生产日期/有效期 | 是否标注 |
| 6 | 执行标准号 | 是否标注 |
| 7 | 备案号 | 如需是否标注 |
| 8 | 警示语 | 如需是否标注 |
| 9 | 自我声明 | 推荐：是否标注"本产品属于日用品，不具有疾病预防、治疗功能，不可替代药品或医疗器械" |

**审核输出**：缺少的要素标🔴，可选的标🟡

**维度C：企标对照**（⚠️ 仅包装类型触发）

1. 问审核人要企标文件：**"这个产品的企标文件有吗？发我对照一下标签内容。"**
2. 审核人发文件 → AI读取后对照
3. 审核人说没有/给不了 → 跳过，告知"缺少企标文件，标签对照未执行"

**对照规则**：
- 通读企标全文，找到标签相关章节（标题通常含"标签""标志"，约第7章附近，不固定）
- **产品名称**：企标要求"包装上必须有产品名称"，包装名 ≠ 企标品类名属正常
- **委托方**（武汉市蔡氏福宁中草药有限公司）与企标起草单位一致即可，受托方可以不同
- 超出企标最低要求的自有信息属加分项
- **企标有效期检查**：找到实施日期/有效期 → 与当前日期对比 → 过期🔴 / 即将过期🟡 / 未找到则注明

#### 3c. 输出审核结果

列出所有风险点，标注等级，给出修改建议。**等审核人确认哪些要改、哪些放行后，再进入步骤4。**

> 输出格式详见 `references/输出模板.md`（步骤3c 审核结果模板 + 步骤4a 报告模板）

### 步骤4：输出与沉淀 — 生成报告 + 知识库回写

**目标**：出正式报告，新确认的放行内容沉淀到知识库

#### 4a. 生成审核报告

**报告模板**：详见 `references/输出模板.md`（步骤4a 报告模板段）

#### 4b. 知识沉淀（企微文档）

**检查是否有新确认的可用风险词**：
- 本次审核中，审核人确认了某个风险词汇"可用"
- 该词汇不在当前法务背书词汇清单中

**有新词 → 询问审核人**：
```
发现"[词汇]"本次确认为可用，是否录入《蔡氏福宁 违禁词白名单》文档？
```

- 审核人确认 → 调用 `wecom-cli doc edit_doc_content` 将新词追加到企微文档
- 审核人拒绝 → 记录到记忆文件，下次提醒

**API调用（企微文档追加）**：

> ⚠️ **三个关键点**：
> 1. `edit_doc_content` 是**覆写**不是追加，必须先读出原内容，拼接新行后再整体写入
> 2. **必须用 Node.js** 构造 JSON，Shell 变量拼接遇到文档中的换行/引号会转义出错
> 3. 读取和写入都是**异步任务**，需要轮询 `task_done`，最多重试 5 次

```bash
cat > /tmp/write_vocab.js << 'SCRIPT_EOF'
const { execSync } = require("child_process");

const DOCID = "dce_tRfn7KasEa5-jt0R9m9-VBhCrZtoyuoAyOFzwzwTdFt2vQ_WMSXYfU8HbfaCtB9h4DWuQtSz-1fzEcaUgAcw";
const NEW_ROW = "| [词汇] | [背书依据] | [YYYY-MM-DD] |";

// 带轮询的读取
function readDoc() {
  let taskId = null;
  for (let i = 0; i < 5; i++) {
    const payload = taskId
      ? { docid: DOCID, type: 2, task_id: taskId }
      : { docid: DOCID, type: 2 };
    const res = JSON.parse(execSync(
      `wecom-cli doc get_doc_content '${JSON.stringify(payload)}'`,
      { encoding: "utf8" }
    ));
    if (!res.task_done) { taskId = res.task_id; continue; }
    return res.result.content;
  }
  throw new Error("读取失败（重试5次后）");
}

// 带轮询的写入
function writeDoc(content) {
  let taskId = null;
  for (let i = 0; i < 5; i++) {
    const payload = taskId
      ? { docid: DOCID, content, content_type: 1, task_id: taskId }
      : { docid: DOCID, content, content_type: 1 };
    const res = JSON.parse(execSync(
      `wecom-cli doc edit_doc_content '${JSON.stringify(payload)}'`,
      { encoding: "utf8" }
    ));
    if (!res.task_done) { taskId = res.task_id; continue; }
    console.log("✅ 写入成功");
    return;
  }
  throw new Error("写入失败（重试5次后）");
}

// 执行：读取 → 拼接 → 写入
const content = readDoc();
writeDoc(content + "\n" + NEW_ROW);
SCRIPT_EOF

# 使用 managed Node 避免版本问题
NODE_BIN=$(which node 2>/dev/null || ls /Users/zankus/.workbuddy/binaries/node/versions/*/bin/node 2>/dev/null | head -1)
$NODE_BIN /tmp/write_vocab.js
rm /tmp/write_vocab.js
```

## 🚫 禁止行为

| # | 禁止 | 原因 |
|---|------|------|
| 1 | 自动跳步（不经审核人确认直接进入下一环节） | 每步确认是防错根基，跳步将导致审核结果基于未确认的中间数据 |
| 2 | 识别失败时自行编造文案内容 | 基于错误内容的审核结论全部作废，浪费时间 |
| 3 | 硬编码法务词汇替代企微文档读取 | 词汇来源必须实时，硬编码会导致审核依据过时 |
| 4 | 保存审核报告到非 workspace 目录（如 /tmp/、隐藏目录） | 文件丢失后需要重审 |
| 5 | 修改企微文档《违禁词白名单》中已有法务背书内容 | 已背书词汇需保留完整的历史记录，只能追加新行 |
| 6 | 在一次回复中同时输出步骤2的核实结果和步骤3的审核结论 | 步骤2确认前不应出现审核判断，防止数据未确认就出结论 |
